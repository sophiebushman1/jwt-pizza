# Incident: 2026-04-07

## Summary

Between approximately 9:00 AM and 1:54 PM on April 7, 2026, all pizza orders through pizza.sophiebyu.click were failing with a 500 error. The incident was caused by chaos intentionally injected at the JWT Pizza Factory level as part of a scheduled chaos testing exercise. The failure was detected at 1:53 PM via manual inspection of the Grafana logs panel, which showed the factory returning `{"message":"chaos monkey"}` on every order. The chaos was resolved at 1:54 PM by navigating to the `followLinkToEndChaos` URL embedded in the failed order response.

## Detection

The incident was detected at 1:53 PM when the Grafana Service Logs panel showed ERROR entries on `POST /api/order` with a 500 status code and a `resBody` containing `"followLinkToEndChaos"` and `"Failed to fulfill order at factory"`. The Factory Service Requests log panel confirmed the factory was returning `{"message":"chaos monkey"}` on all order requests.

The `generatePizzaTraffic.sh` script was running against `https://pizza-service.sophiebyu.click` throughout the chaos window, which caused order requests to continuously hit the factory and produce log evidence of the failure. The Grafana alert rule for pizza failures did not send an email notification during this incident; this is noted as an action item.

## Impact

From approximately 9:00 AM to 1:54 PM (roughly 5 hours), all pizza orders to the JWT Pizza Factory were failing. Users attempting to place orders received a 500 error with the message `Failed to fulfill order at factory`. No successful pizza orders were fulfilled during the chaos window.

## Timeline

All times are Mountain Time (UTC-6).

- _9:00 AM_ - Chaos window opened. `generatePizzaTraffic.sh` started against `https://pizza-service.sophiebyu.click`.
- _9:00 AM – 1:51 PM_ - Traffic script running. No Grafana email alert received despite pizza order failures occurring.
- _1:51 PM_ - Noticed no alert emails had arrived. Opened Grafana dashboard to investigate manually.
- _1:53 PM_ - Grafana Service Logs panel showed `ERROR` on `POST /api/order` (status 500). Factory Service Requests panel confirmed factory returning `{"message":"chaos monkey","reportUrl":"https://pizza-factory.cs329.click/api/support/ab5220e908da432a8d7da03b632a2535/report/kruzimls"}`.
- _1:53 PM_ - Identified `followLinkToEndChaos` URL in the error response: `https://pizza-factory.cs329.click/api/support/ab5220e908da432a8d7da03b632a2535/report/kruzimls`.
- _1:54 PM_ - Navigated to the chaos resolution URL in browser.
- _1:54 PM_ - Factory responded: `{"message":"Problem resolved. Pizza is back on the menu!"}`. Incident resolved.

## Response

I was running the traffic script during the chaos window. At 1:51 PM I checked Grafana after noticing no alert emails had arrived. I:

1. Opened the Grafana Pizza Dashboard and navigated to the Service Logs panel.
2. Observed `ERROR` log entries showing `POST /api/order` returning status 500 with the message `"Failed to fulfill order at factory"`.
3. Checked the Factory Service Requests log panel, which confirmed the factory was responding with `{"message":"chaos monkey"}` and a `reportUrl`.
4. Copied the `followLinkToEndChaos` URL from the log entry:
   ```
   https://pizza-factory.cs329.click/api/support/ab5220e908da432a8d7da03b632a2535/report/kruzimls
   ```
5. Navigated to the URL in a browser tab. The factory confirmed the chaos was ended.

## Root cause

The JWT Pizza Factory injected chaos by returning HTTP 500 with `{"message":"chaos monkey"}` on all pizza order requests. The `orderRouter.js` in jwt-pizza-service correctly surfaced the factory's `reportUrl` as `followLinkToEndChaos` in the error response body. No application code changes were needed — the fix was entirely at the factory level by reporting the chaos via the provided URL.

## Resolution

Navigated to the `followLinkToEndChaos` URL found in the failed order response log:
`https://pizza-factory.cs329.click/api/support/ab5220e908da432a8d7da03b632a2535/report/kruzimls`

The factory confirmed resolution with `{"message":"Problem resolved. Pizza is back on the menu!"}`. The traffic script continued running and orders resumed successfully.

No changes to jwt-pizza-service code were required.

## Prevention

- **Alerting**: The Grafana "Pizza Order Failures" alert rule (`increase(pizza_failures_total[5m]) > 0`) was configured but did not fire during this incident. The email contact point routing should be verified to ensure alerts are delivered in future incidents.
- **Dashboard**: The Grafana pizza dashboard panels were updated to use `rate(...[5m])` instead of `[1m]` prior to this incident, which allowed the logs panels to display correctly during diagnosis.
- **Runbook**: When pizza orders fail, check the Grafana Service Logs for a `followLinkToEndChaos` URL in the `resBody` field. Navigate to that URL immediately to report and end factory-injected chaos.
